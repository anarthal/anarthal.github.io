Resultsets are obtained as a result of a SQL statement execution, by calling connection::query or statement::execute. A resultset holds metadata about the operation (meta), allows you to read the resulting rows ( read_one, read_some, read_all), and stores additional information about the execution ("EOF packet data", in MySQL slang: affected_rows, last_insert_id, and so on). Resultsets are the glue that join the different pieces of connection::query and statement::execute multi-function operations.

Resultsets are proxy I/O objects, meaning that they hold a reference to the internal state of the connection that created them. I/O operations on a resultset result in reads and writes on the connection's stream. A resultset is usable for I/O operations as long as the original connection is alive and open. Moving the connection object doesn't invalidate resultsets pointing into it, but destroying or closing it does. The executor object used by a resultset is always the underlying stream's.

Resultsets are default-constructible and movable, but not copyable. A default constructed or closed resultset has !this->valid(). Calling any member function on an invalid resultset, other than assignment, results in undefined behavior.
